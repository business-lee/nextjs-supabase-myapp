type: feedback
---

역할 기반 커뮤니티/모임 앱 PRD를 작성할 때 다음 패턴이 효과적이었다:

1. **사용자 여정을 역할별로 분리**해서 기술한다. 주최자 여정과 참여자 여정을 각각 단계별로 명시하면 개발자가 권한 분기 로직을 명확히 파악할 수 있다.

2. **데이터 모델에 상태값(status) 전이를 명시**한다. `'pending' | 'approved' | 'rejected' | 'waitlisted'` 처럼 허용 값을 PRD에 직접 적어두면 enum 없이 text 타입을 써도 개발 혼선이 없다.

3. **핵심 상호작용 섹션에 분기 흐름을 의사코드로 작성**한다. 초대 링크 흐름, 참가 신청 승인 흐름, 정산 계산 흐름처럼 복잡한 비즈니스 로직은 텍스트보다 의사코드 블록이 훨씬 명확하다.

4. **RLS 원칙을 데이터 요구사항 섹션에 테이블별로 명시**한다. Supabase 프로젝트에서 RLS는 보안의 핵심이므로 PRD 단계에서 정책 방향을 확정해두어야 나중에 재작업이 없다.

**Why:** MeetUp Manager PRD 작성 시 이 패턴으로 작성했고 solo 개발자가 바로 구현에 착수할 수 있는 수준의 명세가 완성되었다.

**How to apply:** 유사한 역할 기반 앱(스터디 그룹, 팀 관리, 이벤트 플랫폼 등) PRD 작성 시 동일 패턴 적용.
